# Wine Log App - Product Brief

## Core User Flows

1) Onboard & First Log: a) User signs up (email/Social) or continues as limited guest; b) Onboarding nudges user to create first entry and sets privacy default to private; c) User opens 'Log Bottle' -> manual entry form opens; d) User enters wine metadata (name, producer, vintage, region, grapes, price, location); e) User enters tasting notes, rating, tags and saves -> entry appears in personal journal; f) Success toast and prompt to share or log another. 2) Manual Entry & Save: a) From 'Log Bottle' user fills out the wine metadata form; b) User edits fields, adds notes/rating/tags; c) User saves the entry -> entry appears in the journal. 3) Discover Similar Wines: a) User opens an entry -> taps 'Find Similar' -> app queries similarity service (external DB and internal heuristics) and returns ranked matches with short rationale (grape, region, tasting profile); b) User browses results, can add suggestions to wishlist or create new log; c) Click-throughs track recommendations performance. 4) Journal Management & Sharing: a) User views personal journal (list/grid) with search & filters (tags, rating, grape, region, date); b) User can edit/delete entries, bulk-select to export CSV, or toggle entry/share settings (private, shareable link, public); c) If sharing, system generates short-lived or permanent link and preview card. 5) Subscription & Account: a) User visits Billing/Upgrade -> sees free limits and paid tiers; b) User enters payment via Stripe -> subscription activated unlocking premium features (unlimited entries, recommendation engine); c) Billing events and access levels update immediately.


## Product Overview

Wine Log App is a responsive web application for wine lovers to log bottles, record tasting notes and ratings, discover similar wines, and build a private or shareable personal wine journal. It targets enthusiasts who want a fast, beautiful way to capture tasting moments, build a searchable archive of bottles, and get discovery recommendations without exposing their entire collection. The core problem it solves is friction in capturing accurate wine information and retaining tasting context (notes, rating), plus discovering similar wines based on logged items. The MVP focuses on friends/family usage with third‑party wine database integration and a likely subscription monetization model for advanced features.


## Definition Of Done

MVP completion criteria: all MVP features implemented and demoed; unit/integration/E2E tests passing for core flows; performance targets met (average page load <1.5s on mid-tier connection, recommendation latency <3s); security checks completed and critical CVEs remediated; accessibility basics met (WCAG 2.1 AA key flows); analytics instrumentation for core events validated; CI/CD pipeline configured with automated deployments to staging and production; backup and monitoring in place (errors, performance, business metrics); documentation: product spec, runbook, API docs (if any), onboarding docs and support FAQs; legal docs: Terms of Service, Privacy Policy, cookie policy available; billing connected to Stripe sandbox and test scenarios validated; customer support process ready. Sign-off criteria: product lead verifies feature completeness and UX; engineering lead validates stability, tests and infra; security officer approves; legal approves policies; operations approves deployment readiness. Release gating: soft launch to closed group (friends/family) with monitoring for 7–14 days and no critical incidents before open launch.


## Business Rules Logic

Privacy default: new entries default to 'private' unless user changes. Free tier limits: e.g., up to 100 entries lifetime (configurable) or 3 free logs during trial; exceeding prompts upgrade. Share link rules: token unique, optional expiry, password protection allowed; if deleted, link invalidated. Duplicate detection: on save, system checks for existing WineEntry with same wine_record_id + vintage + user_id -> suggest merge or mark duplicate. Rating validation: allowed values 0.5–5.0 in 0.5 increments; notes length max 10,000 chars. Billing constraints: subscription required for unlimited logs and premium recommendations; downgrades restrict features at period end. Data retention & deletion: on account deletion, user data queued for purge within X days and backup snapshots retained per policy; GDPR requests require verification. API rate limiting: per-user and per-IP limits; exceed triggers throttling with informative messaging and upgrade CTA. Moderation: reported public entries queued for moderator review; admins may suspend accounts on repeat violations. Merge rules: merging entries retains most recent notes and concatenates history, keeps oldest created_at and archives duplicates. When rules violated, system shows clear error, suggests remedial actions, and logs the event for audit.


## Data Model Key Objects

User: id, name, email, password_hash, oauth_ids, subscription_tier, preferences, created_at, last_active. WineEntry: id, user_id, wine_record_id (nullable), title, vintage, producer, region, grapes, purchase_date, price, location, notes, rating, tags[], privacy (private/share/public), share_link_id, created_at, updated_at. WineRecord (external canonical): external_id, source_provider, canonical_name, vintage_range, grapes[], region, producer, average_rating, external_metadata_blob. ShareLink: id, wine_entry_id, token, expires_at (nullable), password_hash (nullable), access_count. Recommendation: id, base_wine_entry_id, recommended_wine_record_id, score, rationale, generated_at, source. Tag/Collection: id, user_id, name, entries[]. Subscription: id, user_id, tier, status, stripe_customer_id, stripe_subscription_id, current_period_end. ActivityLog: id, user_id, action_type, target_id, metadata, timestamp. Relationships: User 1..* WineEntry; WineEntry 0..1 WineRecord; WineEntry 0..1 ShareLink; WineEntry *..* Tag via join; Recommendation links entries to WineRecords.


## User Roles Permissions

Admin: full platform access, user management (suspend/delete), view analytics, manage API keys, billing overview, content moderation; cannot view private user entries unless requested and justified (audit logs required). User (Subscriber/Free): create/edit/delete own entries, generate share links, follow users (if social enabled), export/import own data, manage subscription, set privacy, report content. Guest (unauthenticated): browse public shared entries via links, limited trial logging (e.g., 3 entries) stored in ephemeral cookie/session, cannot access recommendation engine or export. Moderator (optional role): review reported content, flag or request removal, limited user account actions. Integration Service Account: scoped API credentials for third‑party integrations with read-only access to matched wine metadata and write access only for their own verified entries. Role constraints: admins can not modify user-owned private content without explicit consent; data deletion requests must follow GDPR flow and are audited.


## Feature Specifications

MVP (must-have): user auth (email + social login), create/edit/delete wine logs with manual metadata entry, tasting notes + rating, private/shareable entries, journal list/grid views, search basic (text), filters (tag, rating, date), subscription billing integration (Stripe) and basic free tier limits, analytics events (signup, create entry, share, upgrade), basic admin dashboard for API keys and user support. V1.1 (post-launch priority): better search (faceted + fuzzy), ability to follow friends and view shared entries, social sharing integrations (Twitter, FB), curated recommendations & editorial content, bulk import/export (CSV), notifications (email/in-app), user collections/queues, basic offline capture queue. V2+ (future roadmap): advanced recommendations (ML personalization), cellar inventory and tracking (bottles left, location), retailer/marketplace links & purchase integration, tasting event management and multi-user journals, richer analytics dashboards for users (taste profile), open API for integrations, team/family shared journals and roles, mobile native apps.


## Testing Qa Requirements

Unit tests: for core domain logic (rating validation, duplicate detection, subscription gating), 80%+ coverage on critical modules. Integration tests: auth flows, third‑party API contract tests (mocked + live staging), billing flows (Stripe test suite), search and recommendation integration. End-to-end (E2E): critical user journeys (signup->first log, manual entry & save, share, upgrade) automated (Cypress/Puppeteer) across major browsers. Performance tests: background job throughput, recommendation latency at scale, page load budgets. Security tests: penetration test pre-launch, dependency vulnerability scans, authentication and authorization checks. Accessibility tests: automated (axe) plus manual tests to meet WCAG 2.1 AA. Cross-platform/responsive: test on major browsers and viewport sizes, mobile devices and tablets. Edge case tests: offline autosave/recovery, API error/fallback behavior, payment failure and proration. Acceptance criteria: all critical flows pass E2E, no show-stopping bugs, performance targets met, security issues remediated, analytics events firing for core events. Beta/staged release: gated rollout to subset of users with monitoring for errors and behavioral metrics before 100% launch.


## Edge Cases Error Handling

Network loss mid-edit: autosave drafts locally (IndexedDB) and sync on reconnect; show offline indicator. Third‑party API rate limit or outage: fall back to cached results and allow manual entry; show banner explaining degraded functionality and ETA. Conflicting edits (multi-device): optimistic locking with conflict detection on save -> show diff and let user choose version or merge. Payment failure: notify user, suspend premium features after grace period, send email with payment retry link. Expired share link: show message and CTA to regenerate. Duplicate detection false positives: allow override with confirmation. Large import failures: validate CSV before import and provide row-level error report. Account deletion misuse: require re‑authentication and cooldown period; log actions. Data corruption: backups and immutable logs allow restore; surface minimal maintenance page and ETA. Security incidents: immediate account lock, inform affected users, follow incident response plan. All errors surfaced to users should be actionable, non-technical, and logged with correlation IDs for support.


## Success Metrics Analytics

Acquisition & Activation: signups per week, percent completing onboarding, time to first log (target < 5 minutes). Engagement: Daily Active Users (DAU), Monthly Active Users (MAU), average entries per user/month, session length. Retention & Cohorts: 7/30/90-day retention, churn rate for subscribers, cohort LTV. Conversion & Revenue: free->paid conversion rate, ARPU by cohort, MRR, churn by tier. Feature usage: percent of logs created via manual entry, percentage using 'Find Similar', share link generation rate, export rate. Quality & Performance: average page load times. Support & Trust: number of reports/moderation actions, time to resolve support tickets. Recommendation effectiveness: click-through rate on recommendations, conversion to new logs from recommendations. Use instrumentation: track events (signup, login, create_entry, edit_entry, delete_entry, manual_entry, find_similar, share_link_created, upgrade_initiated, payment_success, export_csv) with properties (user_id, entry_id, timestamp, referral_source). Define targets and alerting for major regressions (e.g., page load >1.5s).


## Ux Ui Design System Patterns

Design system: mobile-first responsive web with 8pt grid, neutral warm palette (rich burgundy, cream, slate), accessible contrast (WCAG 2.1 AA), primary typeface: modern sans for legibility, secondary serif for headings (optional). Core components: top nav with logo, search, 'Log Bottle' CTA; persistent secondary nav for Filters/Collections; card component for wine entries (name, rating, short note); modal patterns for confirmations; forms using single-column layout, autosave in progress, inline validation and contextual help; toast notifications for success, unobtrusive banners for errors; rating control with half-star support and quick keyboard shortcuts; tag chips with multi-select; share link generator modal with copy button and expiry selector. Interaction patterns: progressive disclosure for advanced metadata, optimistic UI on save, batching for bulk operations, accessible keyboard navigation, clear affordances for privacy toggle. Design references: Vivino (entry flow), Notion (clear cards & editing), Figma component libraries for consistent spacing and tokens. Accessibility: aria labels on interactive elements, focus management for modals, colorblind-friendly palette.


## Technical Constraints Dependencies

Third‑party dependencies: wine DB API (provider contracts and costs), payments (Stripe), hosting (AWS/GCP/Vercel), object storage (S3-compatible), relational DB (Postgres), search engine (Algolia or Elasticsearch), analytics (Segment/Amplitude), email provider (SendGrid/Postmark), CDN. Constraints: external wine DB may have no public API or commercial rate limits — plan for cached local canonical store and contractual integration. Performance: recommendation latency <3s, search <500ms, average page load <1.5s. Security & Compliance: TLS everywhere, encryption at rest for user data, PCI DSS via Stripe, GDPR data deletion/exports. Scalability: design for growth from thousands to millions of entries; use background jobs for heavy tasks (recommendations), queueing (RabbitMQ/SQS), and autoscaling. Platform constraints: web-first responsive, progressive enhancement for offline capture. Cost constraints: third‑party lookups cost per API call — throttle and cache aggressively.
